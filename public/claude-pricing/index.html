Token model assumptions
~18 tokens/diff line (context lines, hunk markers, file headers) · 1,000-token system prompt (cached) + 300-token metadata per round (PR title, description, branch info) · Output: max(1,000, lines × 8) tokens initial review, 600 tokens per follow-up · History: 60% of prior round tokens carried forward.
Architecture tip
For reviews with many rounds, summarise earlier history rather than passing the full raw transcript each time. This keeps input tokens from compounding too aggressively on long-running reviews.
Batch API
Reviews don't need instant turnaround — queuing each round async saves 50% with no quality tradeoff. Ideal for Review Board's push-triggered workflow.
Prompt caching
Cache your review guidelines and code standards system prompt once. The 90% saving on cached input tokens compounds across every round of every review automatically.
